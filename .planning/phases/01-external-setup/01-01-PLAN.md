<task type="checkpoint:human-action" gate="blocking">
  <name>Task 2: Create Slack app + incoming webhook + verify with curl (SETUP-02)</name>
  <what-built>After Task 1: #qodo-skills-releases channel exists. Now create the Slack app and authorize a webhook to post to it, then verify the URL is live.</what-built>
  <action>
    Create the Slack app and incoming webhook at https://api.slack.com/apps:
    1. Click "Create New App" → "From scratch"
    2. App Name: `qodo-skills-releases-bot` (or any descriptive name)
    3. Pick Workspace: select your team workspace → click "Create App"
    4. In the left sidebar, click "Incoming Webhooks"
    5. Toggle "Activate Incoming Webhooks" to ON
    6. Click "Add New Webhook to Workspace"
    7. On the authorization screen, **explicitly verify the dropdown shows `#qodo-skills-releases`** — do not accept the default channel
    8. Click "Allow"
    9. Copy the generated webhook URL (format: `https://hooks.slack.com/services/TXXXXXXXX/BXXXXXXXX/XXXXXXXX`)

    Then verify the webhook is live by running this script in your terminal (replace the placeholder URL):

    ```bash
    WEBHOOK_URL="https://hooks.slack.com/services/TXXXXXXXX/BXXXXXXXX/XXXXXXXX"

    HTTP_STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
      -X POST \
      -H 'Content-type: application/json' \
      --data '{"text":"Phase 1 setup verification — safe to ignore"}' \
      "$WEBHOOK_URL")

    if [ "$HTTP_STATUS" = "200" ]; then
      echo "PASS: Webhook returns HTTP 200"
    else
      echo "FAIL: Webhook returned HTTP $HTTP_STATUS"
    fi
    ```

    Troubleshooting:
    - HTTP 403 / `posting_to_general_channel_denied`: Channel has posting restrictions — delete and re-create `#qodo-skills-releases` with default permissions
    - HTTP 403 / `action_prohibited`: Workspace admin approval required
    - Test message in wrong channel: Revoke the webhook, re-authorize with `#qodo-skills-releases` explicitly selected
  </action>
  <how-to-verify>

<task type="checkpoint:human-action" gate="blocking">
  <name>Task 3: Store SLACK_WEBHOOK_SKILLS_RELEASES_URL as GitHub repository secret + verify (SETUP-03)</name>
  <what-built>After Task 2: webhook URL is verified live. Now store it as a masked repository secret so Phase 2 workflow YAML can reference it as ${{ secrets.SLACK_WEBHOOK_SKILLS_RELEASES_URL }}.</what-built>
  <action>
    Store the webhook URL as a GitHub Actions repository secret using one of these methods:

    Method A — GitHub CLI interactive (recommended — keeps URL out of shell history):
    ```bash
    gh secret set SLACK_WEBHOOK_SKILLS_RELEASES_URL
    # Paste the webhook URL when prompted, press Enter, then Ctrl-D
    ```

    Method B — Pipe from environment variable:
    ```bash
    read -s WEBHOOK_URL   # paste URL here, press Enter
    echo "$WEBHOOK_URL" | gh secret set SLACK_WEBHOOK_SKILLS_RELEASES_URL
    unset WEBHOOK_URL
    ```

    DO NOT use `--body` flag with the raw URL inline — puts the URL in shell history.

    Method C — GitHub UI (if gh CLI unavailable):
    Repository → Settings → Secrets and variables → Actions → New repository secret
    Name: `SLACK_WEBHOOK_SKILLS_RELEASES_URL` (exactly — case-sensitive), Value: paste the webhook URL

    Then run the full Phase 1 verification script:
    ```bash
    echo "=== Phase 1 Verification ==="

    WEBHOOK_URL="<paste-webhook-url-here>"
    HTTP_STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
      -X POST -H 'Content-type: application/json' \
      --data '{"text":"Phase 1 final check — safe to ignore"}' \
      "$WEBHOOK_URL")
    [ "$HTTP_STATUS" = "200" ] \
      && echo "PASS: Webhook returns HTTP 200 (SETUP-02)" \
      || echo "FAIL: Webhook returned $HTTP_STATUS (SETUP-02)"

    gh secret list | grep -q "SLACK_WEBHOOK_SKILLS_RELEASES_URL" \
      && echo "PASS: SLACK_WEBHOOK_SKILLS_RELEASES_URL secret exists (SETUP-03)" \
      || echo "FAIL: SLACK_WEBHOOK_SKILLS_RELEASES_URL not found in gh secret list (SETUP-03)"

    echo "=== Done ==="
    ```

    If `gh secret list` is empty after using the GitHub UI: confirm the secret was created under Repository secrets, not Organization secrets or Environment secrets.
  </action>
  <how-to-verify>
    Both lines of the verification script print `PASS`.
  </how-to-verify>
